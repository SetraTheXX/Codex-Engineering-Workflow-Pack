# v0.1 Pilot Test Plan

Date: 2026-05-26

This plan validates Codex Engineering Workflow Pack v0.1 locally before any production-ready claim.

## Pilot 0: Self-dogfooding On This Repo

### Purpose

Validate the pack against its own repository before using it elsewhere.

### Skills

- `setup-codex-engineering-workflow`
- `zoom-out`
- `handoff`

### Test Input

Use this repo as the target:

```txt
C:\Users\tunca\Desktop\Projeler Ve Planlar\Codex Engineering Workflow Pack
```

Ask Codex to:

1. Run setup workflow for this repo.
2. Zoom out on `.agents/skills/`.
3. Create a handoff for the next validation session.

### Expected Output

- Proposed local docs setup under `docs/agents/`.
- Read-only map of current skill pack structure.
- Handoff document that references existing artifacts without duplicating them.

### Success Criteria

- Setup does not overwrite existing docs without approval.
- Zoom-out does not edit files.
- Handoff is concise and points to relevant docs/skills.
- No secrets, destructive commands, or remote publish actions.

### Failure Mode Notes

- Setup may ask too many questions instead of inferring from repo files.
- Zoom-out may become a generic summary instead of a useful system map.
- Handoff may duplicate full report contents instead of linking paths.

## Pilot 1: Small Or Medium Real Repo

### Purpose

Validate the daily engineering workflow on a real codebase with tests or at least runnable commands.

### Skills

- `diagnose`
- `tdd`
- `to-prd`
- `to-issues`

### Test Input

Choose a small or medium repo with one of:

- a known bug,
- a small feature idea,
- existing test commands,
- clear user-facing behavior.

Run:

1. `to-prd` for the feature or fix plan.
2. `to-issues` to split the plan into vertical slices.
3. `tdd` for one small accepted slice.
4. `diagnose` on a known or simulated failure.

### Expected Output

- Local markdown PRD.
- Local markdown issue files under the repo's configured issue path.
- One red-green-refactor cycle or documented reason it could not run.
- One deterministic diagnosis feedback loop or explicit blocker.

### Success Criteria

- `to-prd` does not start unnecessary interview if context is sufficient.
- `to-issues` avoids backend-only/frontend-only/test-only slices unless justified.
- `tdd` writes one failing test before implementation.
- `diagnose` establishes repro before fix.
- Verification commands are run or blockers are documented.

### Failure Mode Notes

- `to-issues` may still produce slices that are too large.
- `tdd` may struggle if repo has no test harness.
- `diagnose` may need user-provided logs or fixture for non-reproducible bugs.
- PRD may include stale file-level implementation details if not checked.

## Pilot 2: Larger Real Repo

### Purpose

Validate planning, prototyping, architecture audit, and handoff behavior on a more complex codebase.

### Skills

- `grill-with-docs`
- `prototype`
- `improve-codebase-architecture`
- `handoff`

### Test Input

Choose a larger repo with:

- multiple modules or domains,
- docs or ADRs,
- UI or state/business logic uncertainty,
- known architectural friction.

Run:

1. `grill-with-docs` on an ambiguous feature or refactor idea.
2. `prototype` for either UI or logic uncertainty.
3. `improve-codebase-architecture` on one bounded module area.
4. `handoff` after the audit/prototype decision.

### Expected Output

- Clarified plan with domain terms and open decisions.
- Approved throwaway prototype plan or prototype output.
- Local architecture audit report proposal or approved report.
- Handoff that captures decisions, risks, and next skill recommendation.

### Success Criteria

- `grill-with-docs` reads docs/code before asking questions.
- `prototype` clearly marks throwaway scope and exit decision.
- `improve-codebase-architecture` gives file/module-specific, sequenced recommendations.
- No big rewrite recommendation.
- Handoff lets a later Codex session resume without full conversation context.

### Failure Mode Notes

- `grill-with-docs` may over-question instead of reading the repo.
- `prototype` may drift toward production implementation.
- Architecture audit may become too abstract without module references.
- Handoff may omit verification status or open risks.

## Pilot Completion Criteria

The v0.1 pack is ready for broader local use only after:

- Pilot 0 completes on this repo.
- Pilot 1 completes on at least one real repo.
- Pilot 2 completes on one larger repo.
- At least three failure mode notes are reviewed.
- Any blocking skill wording issues are patched.

Do not call v0.1 production-ready until these pilots are complete.
